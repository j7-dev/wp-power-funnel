# Clarify Session 2026-04-01 15:00

## Idea

釐清 Power Funnel WordPress 外掛的工作流引擎完整規格，涵蓋：
1. WorkflowRule 與 Workflow 的生命週期
2. 觸發機制（TriggerPointService 事件橋接）
3. 執行流程（節點逐一執行、狀態轉換）
4. 錯誤處理與遞迴防護（RecursionGuard）
5. 節點系統（BaseNodeDefinition、所有 NodeDefinition、NodeDTO 結構）
6. 節點定義註冊機制（filter hook）
7. 排程機制（WaitNode + ActivitySchedulerService）
8. Context 與參數系統（ParamHelper）

## 原始碼分析結果

### 已實作的 NodeDefinition（execute() 有實作）
- EmailNode: 使用 wp_mail 發送 HTML 郵件，支援訊息模板 (MessageTemplateDTO)
- WaitNode: 使用 as_schedule_single_action 排程延遲，但 timestamp 取得邏輯有 TODO 標記

### 尚未實作的 NodeDefinition（execute() 拋出 BadMethodCallException）
- LineNode
- SmsNode
- WebhookNode
- WaitUntilNode
- TimeWindowNode
- YesNoBranchNode
- SplitBranchNode
- TagUserNode

### ETriggerPoint 枚舉存根（無觸發機制）
- ACTIVITY_ENDED: 無結束時間資料來源
- PROMO_LINK_CLICKED: 無點擊追蹤機制

### 架構特點
- WorkflowDTO::try_execute() 為執行核心，根據 results.length 決定執行哪個節點
- 節點執行後成功不呼叫 do_next()（由 try_execute 迴圈驅動）；跳過（301）時呼叫 do_next()
- WaitNode 執行後不呼叫 do_next()，由 Action Scheduler 到期後重新觸發 running hook
- context 使用 lazy evaluation 模式（context_callable_set 儲存 callable + params，延遲執行）

## Q&A

- Q: 規格涵蓋範圍 -> A: 涵蓋工作流引擎的完整生命週期，從 WorkflowRule 建立到 Workflow 執行完成
- Q: 是否包含前端 UI 規格 -> A: 僅描述後端引擎機制，前端 UI 規格已在 workflow-rules-crud 和 node-definitions-api specs 中
- Q: NodeDefinition 未實作的部分如何處理 -> A: 在規格中標記目前實作狀態，未實作的以 form_fields schema 記錄
